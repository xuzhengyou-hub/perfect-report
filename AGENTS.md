# AGENTS.md

本文定义本仓库中编码代理的工作方式。

## 核心原则

### 规则 1 - 先思考，再编码
- 明确写出假设。
- 不确定时先问，不要猜。
- 有歧义时给出多种理解。
- 如果有更简单的方案，直接指出。
- 一旦困惑，停下来说明不清楚的点。

### 规则 2 - 简单优先
- 只写解决问题所需的最少代码。
- 不加猜测性的功能。
- 不为一次性场景引入抽象。
- 如果资深工程师会觉得复杂，就继续简化。

### 规则 3 - 手术式修改
- 只改任务必需的文件和行。
- 只清理自己改动引入的问题。
- 非必要不要重构、格式化或顺手优化周边代码。
- 保持现有代码风格和本地模式一致。

### 规则 4 - 以目标驱动执行
- 改动前先定义清晰的成功标准。
- 持续迭代，直到标准被验证。
- 不机械走清单，让目标主导过程。

### 规则 5 - 只在需要判断时使用模型
- 把模型用于分类、起草、总结和信息提取。
- 不要把模型用于路由、重试或确定性转换。
- 代码能可靠回答的，就交给代码。

### 规则 6 - Token 预算不是建议
- 单任务预算：4000 tokens。
- 单会话预算：30000 tokens。
- 接近上限时，先总结当前状态，再用新上下文继续。
- 不要默默超预算。

### 规则 7 - 冲突要暴露，不要折中平均
- 模式冲突时，只选一个。
- 优先选择更新或验证更多的方案。
- 说明选择理由。
- 未选方案标记为后续清理项，不要混用。

### 规则 8 - 先读后写
- 加代码前，先看导出、直接调用方和共享工具。
- 不要因为看起来独立就假设它真的独立。
- 结构看不懂时，先停下并问清原因。

### 规则 9 - 测试验证意图，不只验证表象
- 测试要说明这条行为为什么重要。
- 当底层业务规则改变时，测试应该失败。
- 只验证表面行为而不体现意图的测试，不够。

### 规则 10 - 每个重要步骤后都做检查点
- 每完成一个关键步骤，都要总结：
  - 改了什么；
  - 验证了什么；
  - 还剩什么。
- 如果当前状态说不清，就不要继续。
- 一旦失去上下文，先重新描述状态再往下做。

### 规则 11 - 遵守代码库约定
- 即使你在别处会选别的做法，这里也优先遵守仓库约定。
- 在仓库内保持一致，比个人偏好更重要。
- 如果某个约定明显有害，要明确提出，不要悄悄偏离。

### 规则 12 - 明确暴露未完成项
- 重要步骤没做完时，不要声称已完成。
- 没跑的测试，不要说通过了。
- 不确定项、遗漏和阻塞都要清楚说明。

### 规则 13 - 完成的工作必须记录到 ITERATION_LOG
- 任何已完成的功能、增强或修复，都必须记录到 `ITERATION_LOG.md`。
- 记录应包含目标、主要改动、验证结果和剩余问题（如有）。
- 对应迭代记录未更新前，不算完成。

### 规则 14 - 有针对性地使用相关 Skill
- 如果仓库内可用的 skill 适合当前任务，应优先使用。
- 如果某个 skill 与任务高度匹配，就必须使用，不能随意跳过。
- 如果明显该用却没用，要说明原因或阻碍。

### 规则 15 - 代码审查后更新 README
- 每次代码审查完成后，如果审查改变了已记录的行为、流程、验证说明或仓库指南，就更新 `README.md`。
- 不要让 `README.md` 在重要审查后变旧。
- 如果这次审查不需要改 README，也要明确说明。

### 规则 16 - 新会话先读 README 和 ITERATION_LOG
- 新会话开始时，先读 `README.md` 和 `ITERATION_LOG.md`，了解项目现状和最近历史。
- 在做新假设前，把这两个文件当作默认摘要。
- 如果任一文件过期、不完整或不可读，要明确说明并谨慎继续。

### 规则 17 - 如果出现与PRD或者TechDesign不同的技术栈或者流程

- 更新PRD或者TechDesign

在本仓库中工作时，代理应当：
- 提议或修改前先阅读相关代码；
- 先定义成功标准；
- 保持改动范围小且可回退；
- 用有针对性的测试或检查验证结果；
- 在新会话且历史相关时，先读 `README.md` 和 `ITERATION_LOG.md`；
- 每次完成功能或修复后更新 `ITERATION_LOG.md`；
- 相关 skill 合适时优先使用，尤其是强匹配场景；
- 代码审查后如文档应变化，及时更新 `README.md`；
- 准确汇报已完成、已验证、已跳过和仍不确定的内容。

## 推荐响应流程

对于非简单任务，按下面顺序推进：
1. 说明假设和歧义。
2. 定义成功标准。
3. 阅读相关代码路径。
4. 做最小可行改动。
5. 用测试或直接检查验证。
6. 总结结果和剩余风险。
